Install

Install the package from NPM:

npm install koilib

You can also load it directly to the browser by downloading the bundle file located at dist/koinos.min.js, or its non-minified version dist/koinos.js (useful for debugging).

Node JS

With Typescript import the library:

import {
  Signer,
  Contract,
  Provider,
  Transaction,
  Serializer,
  utils,
} from "koilib";

With Javascript import the library with require:

const {
  Signer,
  Contract,
  Provider,
  Transaction,
  Serializer,
  utils,
} = require("koilib");

Principal Classes

  • Signer: Class containing the private key. It is used to sign.
  • Provider: Class to connect with the RPC node.
  • Contract: Class defining the contract to interact with.
  • Transaction: Class to create transactions.
  • Serializer: Class with the protocol buffers definitions to serialize/deserialize data types.

Examples

Send tokens, get balance

The following code shows how to sign a transaction, broadcast a transaction, and read contracts.

(async () => {
  // define signer, provider, and contract
  const provider = new Provider(["http://api.koinos.io"]);
  // WARNING: Replace "KwkAq..." with a valid WIF. Handle private keys securely.
  const signer = Signer.fromWif("KwkAq..."); 
  signer.provider = provider;
  const koinContract = new Contract({
    id: "15DJN4a8SgrbGhhGksSBASiSYjGnMU8dGL", // KOIN Contract ID
    abi: utils.tokenAbi,
    provider,
    signer,
  });
  const koin = koinContract.functions;

  // Transfer
  const { transaction, receipt } = await koin.transfer({
    from: signer.getAddress(),
    to: "172AB1FgCsYrRAW5cwQ8KjadgxofvgPFd6", // Example recipient
    value: "1012345678", // 10.12345678 koin (assuming 8 decimals)
  });
  console.log(`Transaction id ${transaction.id} submitted. Receipt:`);
  console.log(receipt);

  // wait to be mined
  const { blockNumber } = await transaction.wait();
  console.log(`Transaction mined. Block number: ${blockNumber}`);

  // read the balance
  const { result } = await koin.balanceOf({
    owner: signer.getAddress(),
  });
  console.log('Balance:', result);
})();

Upload contract

It's also possible to upload contracts using koilib. You need the wasm file for that. There are several tools that can help you to develop smart contracts on Koinos:

  • Arkinos: Complete tool to develop contracts in Assembly Script and bootstrap a website to interact with them.
  • koinos/sdk-as-cli: SDK to develop contracts in Assembly Script.
  • koinos-sdk-cpp: SDK to develop contracts in C++.
  • koinosbox/contracts: Contract examples created in Assembly Script. You can also import them in your project by using the npm package.

Here is an example on how deploy a contract:

const fs = require("fs"); // Make sure to import fs

(async () => {
  // define signer, provider and bytecode
  const provider = new Provider(["http://api.koinos.io"]);
  // WARNING: Replace "KwkAq..." with a valid WIF. Handle private keys securely.
  const signer = Signer.fromWif("KwkAq..."); 
  signer.provider = provider;
  const bytecode = fs.readFileSync("my_contract.wasm"); // Make sure 'my_contract.wasm' exists

  // create contract and deploy
  const contract = new Contract({ signer, provider, bytecode });
  const { transaction, receipt } = await contract.deploy();
  console.log("Transaction submitted. Receipt:");
  console.log(receipt);
  
  // wait to be mined
  const { blockNumber } = await transaction.wait();
  console.log(`Contract uploaded in block number ${blockNumber}. Contract ID: ${contract.getId()}`); 
})();

You can also upload a contract in a new address. It is not required that this new address has funds, you just have to set your principal wallet as payer.

const fs = require("fs"); // Make sure to import fs

(async () => {
  // define signer, provider and bytecode
  const provider = new Provider(["http://api.koinos.io"]);
  // WARNING: Replace WIFs. Handle private keys securely.
  const accountWithFunds = Signer.fromWif("K... this account has funds"); 
  const newAccount = Signer.fromWif("L... new account without funds");
  accountWithFunds.provider = provider;
  newAccount.provider = provider; // Set provider for new account too

  const bytecode = fs.readFileSync("my_contract.wasm"); // Make sure 'my_contract.wasm' exists

  // create contract. Set newAccount as signer
  const contract = new Contract({
    signer: newAccount,
    provider,
    bytecode,
    options: {
      // transaction options
      // set payer
      payer: accountWithFunds.address,

      // use "beforeSend" function to sign
      // the transaction with the payer
      beforeSend: async (tx) => {
        await accountWithFunds.signTransaction(tx); // Use await for async operation
      },
    },
  });

  // call deploy()
  // By default it is signed by "newAccount". But, as
  // in beforeSend it is signed by the payer then it
  // will have 2 signatures
  const { transaction, receipt } = await contract.deploy(); // Get transaction object
  console.log("Transaction submitted. Receipt: ");
  console.log(receipt);
  const { blockNumber } = await transaction.wait(); // Wait for the transaction
  console.log(`Contract uploaded in block number ${blockNumber}. Contract ID: ${contract.getId()}`);
})();

In fact, there are several ways to set a different payer and use it to upload a contract. This is another example:

const fs = require("fs"); // Make sure to import fs

(async () => {
  // define signer, provider and bytecode
  const provider = new Provider(["http://api.koinos.io"]);
  // WARNING: Replace WIFs. Handle private keys securely.
  const accountWithFunds = Signer.fromWif("K... this account has funds");
  const newAccount = Signer.fromWif("L... new account without funds");
  accountWithFunds.provider = provider;
  newAccount.provider = provider; // Set provider for new account too

  const bytecode = fs.readFileSync("my_contract.wasm"); // Make sure 'my_contract.wasm' exists

  // create contract. Set newAccount as signer
  const contract = new Contract({
    signer: newAccount,
    provider,
    bytecode,
  });

  // call deploy but do not broadcast the transaction.
  // Also set the payer
  const { transaction } = await contract.deploy({
    payer: accountWithFunds.address,
    sendTransaction: false,
  });

  // sign the transaction with the payer
  await accountWithFunds.signTransaction(transaction);

  // at this point the transaction will have 2 signatures:
  // - signature of newAccount
  // - signature of accountWithFunds

  // now broadcast the transaction to deploy
  // Note: Sending with newAccount since it's the contract owner/signer.
  const { receipt } = await newAccount.sendTransaction(transaction); 
  console.log("Transaction submitted. Receipt: ");
  console.log(receipt);
  const { blockNumber } = await transaction.wait();
  console.log(`Contract uploaded in block number ${blockNumber}. Contract ID: ${contract.getId()}`);
})();

Multisignatures

It can be configured to sign a single transaction with multiple accounts. Here is an example:

// Assuming koinContract and koin (functions interface) are defined as in previous examples
// And signer (the initial signer associated with the contract object) is also defined.

// WARNING: Replace WIFs. Handle private keys securely.
const signer2 = Signer.fromWif("KzP1..."); 
const signer3 = Signer.fromWif("L1t...");
// Signers used for multisig also need a provider if they interact directly (like paying)
signer2.provider = provider; 
signer3.provider = provider;

const addMoreSignatures = async (tx) => {
  // Ensure the transaction has the payer's signature if needed
  await signer2.signTransaction(tx); 
  // Add other required signatures
  await signer3.signTransaction(tx); 
};

// Assuming 'signer' is the primary signer (e.g., the 'from' account or contract owner)
// 'signer2' will be the payer in this example.
const { transaction, receipt } = await koin.transfer( // Make sure 'koin' is defined
  {
    from: signer.getAddress(), // Example: using primary signer's address as 'from'
    to: "1Gvqdo9if6v6tFomEuTuMWP1D7H7U9yksb",
    value: "1000000", // 1 Koin assuming 8 decimals
  },
  {
    payer: signer2.getAddress(), // Set signer2 as the payer
    beforeSend: addMoreSignatures, // Add signatures from signer2 and signer3 before sending
  }
);

console.log('Multisig Transaction submitted:', transaction.id);
console.log('Receipt:', receipt);
await transaction.wait();
console.log('Multisig Transaction mined');

Multiple operations in a single transaction

There are 3 ways to define a transaction with several operations. Use one of them as it is more convenient for you.

  1. Using the Contract class: You can use previousOperations or nextOperations to attach more operations in the transaction:

    // Assuming 'contract' and 'signer' are defined
const fn = contract.functions;
contract.options.onlyOperation = true; // Get only operation object, not full transaction response

// Create operations first
const { operation: op1 } = await fn.set_owner({ // Replace with actual contract methods
  account: "16MT1VQFgsVxEfJrSGinrA5buiqBsN5ViJ",
});
const { operation: op3 } = await fn.set_address({ // Replace with actual contract methods
  account: "1Gvqdo9if6v6tFomEuTuMWP1D7H7U9yksb",
});

// Send the middle operation, attaching the others
contract.options.onlyOperation = false; // Reset to get full transaction response
const { transaction, receipt } = await fn.set_name( // Replace with actual contract methods
  {
    name: "test",
  },
  {
    previousOperations: [op1],
    nextOperations: [op3],
  }
);

console.log('Multi-op Transaction (Method 1) submitted:', transaction.id);
await transaction.wait();
console.log('Multi-op Transaction (Method 1) mined');

  2. Using the Signer class with the Contract class: First all operations are created and then passed to the signer to create and send the transaction:

    // Assuming 'contract', 'signer', and 'provider' are defined
const fn = contract.functions;
contract.options.onlyOperation = true; // Get only operation objects

// Create all operations

  account: "16MT1VQFgsVxEfJrSGinrA5buiqBsN5ViJ",

Estimate mana

Here is an example to estimate the mana. First create a transaction and sign it:

// Assuming 'koin' (contract functions), 'signer', and 'provider' are defined
const tx = new Transaction({ signer, provider });

// Push the operation you want to estimate mana for
await tx.pushOperation(koin.transfer, {
  from: "1NRYHBYr9qxYQAeVqfdSvyjJemRQ4qD3Mt", // Example addresses
  to: "13UdKjYuzfBYbB6bGLQkUN9DJRFPCmG1mU",
  value: "1000", // Example value (smallest unit)
});

await tx.prepare(); // Prepare the transaction (fetches nonce, calculates IDs)
await tx.sign();    // Sign it

// Now send it with broadcast:false. With this option the rpc node
// will compute the transaction but it will not broadcast it, then it will
// not be included in a block:
let { receipt } = await tx.send({ broadcast: false });
console.log('Estimated RC used:', receipt.rc_used);

// Finally, adjust the mana by taking a look to the rc used.
// In this example we will adjust it by increasing it 10%.
// Then sign and send with the default option (broadcast:true):
tx.setRcLimit(Math.round(1.1 * Number(receipt.rc_used)).toString()); // Set adjusted RC Limit as string
await tx.prepare(); // Re-prepare might be needed if headers changed
await tx.sign(); // Re-sign with the new RC limit
receipt = (await tx.send()).receipt; // Send for real (broadcast: true is default)

console.log('Transaction sent with adjusted RC. Receipt:', receipt);
await tx.wait();
console.log('Transaction mined.');

Create ABIs

ABIs are composed of 2 elements: methods and types.

  • The methods define the names of the entries of the smart contract, the corresponding endpoints and the name of the types used.
  • The types all the description to serialize and deserialize using proto buffers.

To generate the types is necessary to use the dependency protobufjs. The following example shows how to generate the protobuf descriptor from a .proto file.

const fs = require("fs");
const pbjs = require("protobufjs-cli/pbjs"); // You might need to install protobufjs-cli

pbjs.main(["--target", "json", "./token.proto"], (err, output) => { // Ensure token.proto exists
  if (err) throw err;
  fs.writeFileSync("./token-proto.json", output);
  console.log("token-proto.json created successfully.");
});

Then this descriptor can be loaded to define the ABI:

const tokenJson = require("./token-proto.json"); // Make sure the file exists

const abiToken = {
  methods: {
    // Example for Koin token - adjust entry_points and types for your contract
    balanceOf: { 
      entry_point: 0x5c721497, // Entry point for balanceOf on Koin contract
      argument: "koinos.contracts.token.balance_of_arguments", // Fully qualified protobuf type name
      return: "koinos.contracts.token.balance_of_result",    // Fully qualified protobuf type name
      read_only: true,
      default_output: { value: "0" } // Optional: Default value if read fails
    },
    transfer: {
      entry_point: 0x27f576ca, // Entry point for transfer on Koin contract
      argument: "koinos.contracts.token.transfer_arguments", // Fully qualified protobuf type name
      return: "koinos.contracts.token.transfer_result",   // Fully qualified protobuf type name
      // read_only: false is default
    },
    mint: { // Assuming a mint function exists
      entry_point: 0xdc6f17bb, // Example entry point
      argument: "koinos.contracts.token.mint_arguments", // Fully qualified protobuf type name
      return: "koinos.contracts.token.mint_result",    // Fully qualified protobuf type name
    },
    // ... other methods
  },
  // Use koilib_types for koilib >= v6, or types for older versions
  koilib_types: tokenJson, // Use the loaded JSON descriptor
};

// You can now use this abiToken when instantiating a Contract
// const customTokenContract = new Contract({ id: 'YOUR_CONTRACT_ID', abi: abiToken, provider, signer });

API Reference / Utilities

This section details some of the utility functions and standard ABIs available in Koilib.

Standard ABIs

Koilib provides standard ABIs for common contract types within utils.

ABI for NFTs (utils.nftAbi)

A generic ABI suitable for contracts following common NFT standards.

Example
import { Contract, Provider, utils } from "koilib";

(async () => {
  const provider = new Provider(["https://api.koinos.io"]); // Use correct endpoint array
  const nicknamesContract = new Contract({
    id: "1KD9Es7LBBjA1FY3ViCgQJ7e6WH1ipKbhz", // Koinos Nicknames Contract
    provider,
    abi: utils.nftAbi, // Using the generic NFT ABI
  });
  const nicknames = nicknamesContract.functions;

  // get the address linked to the nickname "pob"
  // Nicknames uses the nickname string itself as the token ID
  const pobId = `0x${utils.toHexString(new TextEncoder().encode("pob"))}`; 
  const { result } = await nicknames.ownerOf({ token_id: pobId }); // Standard NFT function name
  console.log(result);

  // Expected Output: { value: '159myq5YUhhoVWu3wsHKHiJYKPKGUrGiyv' } 
})();

Formatting and Parsing Units

Function parseUnits

parseUnits(value: string, decimals: number): string

Converts a decimal string representation of a token amount into its integer representation (as a string) based on the specified number of decimals.

Example
import { utils } from "koilib"; 
const { parseUnits } = utils;

const amount = parseUnits("0.00123456", 8);
console.log(amount);
// Output: '123456'
Parameters

value: string - The decimal number string to convert.

decimals: number - The number of decimal places the token uses.

Returns

string - The integer representation as a string.

Function formatUnits

formatUnits(value: string | number | bigint, decimals: number): string

Converts an integer representation (base units, provided as string, number, or bigint) of a token amount into its user-friendly decimal string representation.

Example
import { utils } from "koilib";
const { formatUnits } = utils;

const amount = formatUnits("123456", 8);
console.log(amount);
// Output: '0.00123456'

const amountBigInt = formatUnits(123456n, 8); // Using BigInt
console.log(amountBigInt);
// Output: '0.00123456'
Parameters

value: string | number | bigint - The integer amount in base units.

decimals: number - The number of decimal places the token uses.

Returns

string - The formatted decimal string representation.

Address and Key Encoding/Decoding

Function bitcoinAddress

bitcoinAddress(publicKey: Uint8Array): string

Computes a Koinos address from a public key using the Bitcoin address format (Base58Check encoding of the RIPEMD160 hash of the SHA256 hash of the public key).

Derivation: address = bitcoinEncode( ripemd160 ( sha256 ( publicKey ) ) )

Parameters

publicKey: Uint8Array - The input public key.

Returns

string - The resulting Koinos address.

Function bitcoinEncode

bitcoinEncode(buffer: Uint8Array, type: "public" | "private", compressed?: boolean): string

Encodes a buffer (public or private key) into Bitcoin's Base58Check format. For private keys, this is known as Wallet Import Format (WIF).

See: Bitcoin Base58Check encoding and Bitcoin WIF.

Parameters

buffer: Uint8Array - The key data to encode.

type: "public" | "private" - Specifies the key type.

compressed: boolean = false - Optional (defaults to false), indicates if the key format is compressed (relevant for WIF and some public key formats).

Returns

string - The Base58Check encoded string.

Genesis Data Handling

Function decodeGenesisData

decodeGenesisData(genesisData: GenesisDataEncoded, dictionary?: DictionaryGenesisData): Promise<GenesisDataDecoded>

Asynchronously decodes genesis data (used for launching blockchains) from its Protobuf-encoded format. It uses a dictionary mapping keys to Protobuf types for deserialization. A default dictionary is included for common system keys:

  • "object_key::head_block"
  • "object_key::chain_id"
  • "object_key::genesis_key"
  • "object_key::resource_limit_data"
  • "object_key::max_account_resources"
  • "object_key::protocol_descriptor"
  • "object_key::compute_bandwidth_registry"
  • "object_key::block_hash_code"
Example (Default Dictionary)
import { utils, Serializer } from "koilib"; // Assuming types like GenesisDataEncoded are available
const { decodeGenesisData } = utils;

const genesisData = { // Example GenesisDataEncoded
  entries: [
    {
      space: { system: true },
      // Base64 encoded key for "object_key::genesis_key"
      key: "EiC3nO+XbeKg4C8ugW7M7XdfmJKY4i3l91KoJWxosQPImA==", 
      // Base64 encoded value (protobuf encoded address 1KSQWDyUnFZ48Pf2hsW8Akh1b5fKUWc8Z3)
      value: "AMpASH7CjUHBpl2QR8E5lGKVjVLAvJRg5g==", 
    },
  ],
};

(async () => {

FAQ

Can this library be used to create smart contracts?
No. There are other tools for that. Like Arkinos or koinos/sdk-as-cli for Assembly Script, or koinos-sdk for C++.
Can this library be used to deploy smart contracts?
Yes. If you already have the contract compiled as a .wasm file you can use the Contract class to load the bytecode and deploy it.
Can this library be used to create the ABI for any smart contract?
For the ABI you need the .proto file and the library protobufjs. Then follow the format for the ABI as described in the previous section. It's important to note that this ABI has some differences with respect to the ABI used in koinos-cli. In particular, koilib takes the descriptor from koilib_types (or types in older versions), which is a descriptor in json format, while the ABI in koinos-cli takes the descriptor from types, which is a descriptor in binary format. There are also some differences in the format of "entry point" and "read only".
Can this library be used to interact with smart contracts?
Yes. You can use it to call read_only functions, or send transactions to the contract by calling write functions.
How to generate random keys?
To generate random mnemonic phrases and private keys use @koinosbox/hdKoinos.

Interface Abi

Application Binary Interface (ABI)

ABIs are composed of 3 elements: methods, events, and types.

  • The methods define the names of the entries of the smart contract, the corresponding endpoints and the name of the types used.
  • The events define possible events triggered by the smart contract and the name of the types used.
  • The types contain the description to serialize and deserialize data using proto buffers. It is used to encode/decode the methods and events. These types can be provided in binary format or json format (koilib_types).

To generate the types is necessary to use the dependency protobufjs. The following example shows how to generate the protobuf descriptor from a .proto file.


const fs = require("fs");
const pbjs = require("protobufjs/cli/pbjs");

pbjs.main(
  ["--target", "json", "./token.proto"], // Your proto file path
  (err, output) => {
    if (err) throw err;
    fs.writeFileSync("./token-proto.json", output); // Output JSON descriptor
  }
);

Then this descriptor can be loaded to define the ABI:


const tokenJson = require("./token-proto.json"); // Load the generated descriptor

const abiToken = {
  methods: {
    balanceOf: {
      entry_point: 0x15619248,
      argument: "balance_of_arguments", // Type name from proto
      return: "balance_of_result",      // Type name from proto
      read_only: true,
      default_output: { value: "0" },    // Optional default if contract returns empty
    },
    transfer: {
      entry_point: 0x62efa292,
      argument: "transfer_arguments",
      return: "transfer_result",
    },
    mint: {
      entry_point: 0xc2f82bdc,
      argument: "mint_arguments",
      return: "mint_result",
    },
  },
  events: {
    'koinos.contracts.token.mint_event': { // Event name (can be custom or proto path)
      type: "mint_arguments"              // Type name for event data
    },
    'transfer_event': { // Example custom event name
      type: "transfer_event_data" // Assumes 'transfer_event_data' type exists in proto
    }
  },
  koilib_types: tokenJson, // The loaded JSON descriptor
};

Note that this example uses "default_output" for the method "balanceOf". This is used when the smart contract returns an empty response (for instance when there are no balance records for a specific address) and you require a default output in such cases.

Definition of events

There are 2 ways to define events in koinos:

  1. Event names as protobuffer names: The name of the event directly matches the fully qualified protobuffer message name. In this case, there is no need to define the event explicitly in the ABI's events section.
  2. Custom event names: The contract emits an event with a custom name string. This custom name must then be mapped to its corresponding protobuffer message type within the ABI's events section. This allows reusing protobuffer message types for different events.
